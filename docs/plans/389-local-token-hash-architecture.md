# 389 Local Token Hash Architecture

## Context

`docs/ilonasin-architecture.md` still lists "Should local API tokens be
one-way hashed in SQLite?" as an open question. Current code already answers
yes:

- `internal/credentials/tokens.go` generates `iln_` local client tokens,
  hashes them with `HashToken`, and verifies bearer auth by hash.
- `internal/storage/sqlite/local_tokens.go` stores and queries
  `client_tokens.token_hash`; list/snapshot paths expose only safe prefix and
  last-four fragments.
- Management create returns the full token only once in
  `CreateLocalTokenResponse`; list/snapshot surfaces and the TUI expose
  metadata and safe fragments only.

The architecture doc should describe that settled contract instead of carrying
it as open design debt.

## Goal

Update the architecture document so local API token storage is explicitly
one-way hash based, while preserving current runtime behavior.

## Scope

1. Update the Local API Auth section in `docs/ilonasin-architecture.md`:
   - local client tokens are generated by ilonasin and returned in full only by
     the management create response;
   - SQLite stores a one-way token hash plus safe display fragments, not the
     full token;
   - bearer verification hashes the presented token and compares it with the
     stored hash;
   - management list/snapshot surfaces and the TUI expose only metadata and
     safe fragments.
2. Remove the stale open question about whether local API tokens should be
   one-way hashed.
3. Do not change code, schema, API response shapes, management behavior, TUI
   rendering, logging, or token generation.
